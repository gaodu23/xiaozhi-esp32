# Slot ID 自动探测机制说明

lilygo_t2can这是单独的一个项目。学习他的Slot ID 自动探测功能，把本项目固定的固定 0x02000400改为能学习的。把本项目的摇杆速度喇叭电池电量等帧ID变成学习到的。灯光的先不变。


## 背景：为什么需要探测 Slot ID？

R-Net 总线上的摇杆帧 ID 遵循一个固定模式，但其中包含一个 **Slot 编号**字段，不同轮椅配置中该编号可能不同。例如，默认 Slot 1 的帧 ID 是 `0x02000100`，而某些配置可能使用 `0x02010100`（Slot 2）或其他变体。

如果直接硬编码固定 ID 注入，在与总线实际使用的 Slot ID 不一致时，控制帧会被忽略，轮椅无法响应。

**自动探测的目的：在启动时先"听"总线，找到 JSM 摇杆实际使用的帧 ID，再用这个实际 ID 发送控制帧。**

---

## R-Net 摇杆帧 ID 的结构

以 29-bit 扩展帧 ID 为例，摇杆帧 ID 的字节分布（高位在左）：

```
位  [28..24] [23..16] [15..8]  [7..0]
     0x02      Slot     0x01    0x00
```

| 字节 | 固定值 | 含义 |
|------|--------|------|
| 最高字节 bits 24-27 | `0x2` | 固定标识（摇杆帧类型） |
| 第 3 字节 bits 16-23 | 可变 | **Slot 编号**（如 0x00 = Slot 1, 0x01 = Slot 2 …）|
| 第 2 字节 bits 8-15  | `0x01` | 固定（子协议标识）|
| 最低字节 bits 0-7   | `0x00` | 固定 |

因此合法摇杆帧 ID 示例：

| Slot | CAN ID |
|------|--------|
| 1（默认）| `0x02000100` |
| 2 | `0x02010100` |
| 3 | `0x02020100` |

## 探测流程（代码逻辑）

```
setup():
  调用 detect_joy_frame_id()
    │
    ├─ 开始监听 CAN 总线，超时 2000 ms
    │
    ├─ 对每一帧检查：(frame_id & 0x0F000000) == 0x02000000 ?
    │       ↑
    │   掩码 0x0F000000 只检查 bits 24-27 是否为 0x2
    │   这能匹配所有 Slot 号的摇杆帧
    │
    ├─ 匹配成功 → 返回该帧的完整 identifier（含实际 Slot 编号）
    │
    └─ 2 秒内未见匹配帧 → 回退到默认值 RNET_JOY_ID = 0x02000100
```

探测到的 ID 存入全局变量 `g_joy_frame_id`，后续所有控制帧均以此 ID 发送。

## 过滤器掩码详解

### 探测阶段（宽松匹配）

```
filter = 0x02000000
mask   = 0x0F000000
```

匹配条件：`(frame_id & 0x0F000000) == 0x02000000`

```
掩码二进制：0000 1111 | 0000 0000 | 0000 0000 | 0000 0000
过滤器     ：0000 0010 | xxxxxxxx | xxxxxxxx | xxxxxxxx
                 ↑
         只要 bits 24-27 = 0b0010 即匹配
         Slot 字段、低字节全部忽略 → 匹配所有 Slot
```

### FollowJSM 阶段（精确匹配）

```
filter = RNET_JOY_BASE = 0x02000100
mask   = 0x0F000F00    = 0x0F000F00
```

匹配条件：`(frame_id & 0x0F000F00) == 0x02000100`

```
掩码二进制：0000 1111 | 0000 0000 | 0000 1111 | 0000 0000
过滤器     ：0000 0010 | xxxxxxxx | 0000 0001 | xxxxxxxx
                 ↑                      ↑
         bits 24-27 = 0x2         bits 8-11 = 0x1
         两处需同时满足 → 排除非摇杆帧干扰
```

## 探测流程时序图

```
启动
  │
  ├─ [0 ms]  开始监听，设置 deadline = now + 2000 ms
  │
  │   CAN 总线上的帧流（示例）：
  │   t= 50 ms : 0x1C0C0100  电池帧   → bits 24-27 = 0x1 ≠ 0x2  ✗
  │   t=120 ms : 0x03C30F0F  JSM心跳  → bits 24-27 = 0x0 ≠ 0x2  ✗
  │   t=180 ms : 0x02000100  摇杆帧   → bits 24-27 = 0x2 == 0x2  ✓
  │
  ├─ [180 ms] 匹配！返回 0x02000100，探测结束
  │
  └─ g_joy_frame_id = 0x02000100   后续所有控制帧用此 ID
```

如果轮椅使用 Slot 2：

```
  │   t=180 ms : 0x02010100  摇杆帧  → bits 24-27 = 0x2 ✓
  │
  └─ g_joy_frame_id = 0x02010100   ← 自动适应！
```

## 探测失败的回退策略

若 2 秒内总线上没有出现任何摇杆帧（例如总线尚未就绪，或轮椅尚未开机），则：

```cpp
return RNET_JOY_ID;  // = 0x02000100 (Slot 1 默认值)
```

这是一个安全的默认值：大多数 R-Net 轮椅使用 Slot 1。若确实不匹配，可在 `config.h` 中手动修改 `RNET_JOY_ID`。

## 要点总结

| 环节 | 关键参数 | 作用 |
|------|---------|------|
| 探测超时 | 2000 ms | 给总线足够时间出现摇杆帧 |
| 探测掩码 | `0x0F000000` | 宽松匹配，忽略 Slot 号 |
| FollowJSM 掩码 | `0x0F000F00` | 精确匹配，避免误触发 |
| 结果存储 | `g_joy_frame_id` | 全局唯一，贯穿整个运行周期 |
| 默认回退 | `0x02000100` | Slot 1，适用大多数设备 |
