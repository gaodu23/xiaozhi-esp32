# R-Net CAN 协议与控制模式说明

## 1. CAN 总线基本参数

| 参数 | 值 |
|------|-----|
| 波特率 | 125 kbps |
| 帧类型 | 扩展帧（29-bit ID） |
| TX GPIO | 27 |
| RX GPIO | 26 |

---

## 2. R-Net CAN 帧 ID 一览

| 功能 | CAN ID（扩展 29-bit） | 数据 | 说明 |
|------|----------------------|------|------|
| 摇杆（Slot 1 默认）| `0x02000100` | 见下节 | 方向控制 |
| 摇杆（通用掩码）| `& 0x0F000F00 == 0x02000100` | — | 匹配任意 Slot 摇杆帧 |
| 设置最大速度 | `0x0A040100` | `[pct]` | `pct` = 0x00–0x64（0–100%） |
| 喇叭 ON | `0x0C040100` | 无数据 | `RNET_HORN_ON` |
| 喇叭 OFF | `0x0C040101` | 无数据 | `RNET_HORN_OFF` |
| 左转向灯 | `0x0C040200` | 无数据 | `RNET_IND_LEFT` |
| 右转向灯 | `0x0C040300` | 无数据 | `RNET_IND_RIGHT` |
| 双闪 | `0x0C040400` | 无数据 | `RNET_HAZARD` |
| 灯光 | `0x0C040500` | 无数据 | `RNET_LIGHTS` |
| JSM 心跳 | `0x03C30F0F` | — | JSMerror 模式触发前监听 |
| JSM 错误触发 | `0x0C000000` | 无数据 | 连发 3 次触发错误 |
| 电池电量 | `& 0x0F0F0F00 == 0x1C0C0100` | `[byte0]` = 0–100 | 百分比 |
| 电机电流 | `& 0x0FFFF000 == 0x14300000` | `[byte0,1]` little-endian int16 | 原始ADC |

---

## 3. 摇杆帧格式

帧 ID：`0x02000100`（或总线上自动探测到的实际 Slot ID）  
DLC：4 字节

```
byte[0] = X 轴（左右）  ← ps4_lx()  经 rnet_scale_axis 转换
byte[1] = Y 轴（前后）  ← ps4_ly()  经 rnet_scale_axis 转换（Y 轴取反）
byte[2] = 0x00
byte[3] = 0x00
```

### 轴值编码规则（`rnet_scale_axis`）

PS4 摇杆值范围：`-128 … 127`  
R-Net 编码：将有符号字节直接 cast 为 `uint8_t`  
- 中心 / 死区（±8 内）→ `0x00`
- 正方向（前/右）→ `0x01 … 0x7F`
- 负方向（后/左）→ `0x81 … 0xFF`（补码）
- Y 轴额外取反：摇杆前推（正值）→ R-Net 正向前进

| 摇杆位置 | PS4 值 | byte 值 |
|---------|--------|---------|
| 中心（死区）| -8 … 8 | `0x00` |
| 最大前进 | +127 | `0x7F` |
| 最大后退 | -128 | `0x80` |
| 最大右转 | +127 | `0x7F` |
| 最大左转 | -128 | `0x80` |

## 4. 控制模式

### 4.1 FollowJSM（模式 1，默认）

**原理**：被动跟随总线节奏，在原有 JSM 摇杆帧出现后立即注入自己的帧，与总线时序保持一致。

**流程**：

```
loop():
  1. 等待总线上出现摇杆帧  (ID & 0x0F000F00 == 0x02000100)
     - 超时：FOLLOW_JSM_TIMEOUT_MS = 50 ms
  2. 帧到达 → 立即发送自己的摇杆帧（当前 PS4 摇杆值）
  3. 若超时未见 JSM 帧 → 发送中心（停止）帧保持总线存活
```

**优点**：最小侵入，不破坏 JSM 授权流程。  
**缺点**：依赖 JSM 持续发帧；若 JSM 停发则控制延迟升至 50 ms。

---

### 4.2 JSMerror（模式 2）

**原理**：主动破坏 JSM 的正常运行，让控制器进入错误状态，之后自行独立定时发送摇杆帧接管控制权。

**启动序列**（在 `setup()` 中执行，仅一次）：

```
1. 等待 JSM 心跳帧 (0x03C30F0F) 出现 → 确认 JSM 在线
2. 延迟 200 ms
3. 连续发送 3 × "0C000000#"（每帧间隔 1 ms）→ 触发 JSM 错误状态
4. 延迟 50 ms → 等待 JSM 进入错误状态
```

**主循环**：

```
loop():
  每 10 ms 发送一次摇杆帧（JOY_FRAME_INTERVAL_MS = 10）
  同时不断 drain 接收队列以获取最新遥测数据
```

**优点**：完全自主，不依赖 JSM 帧节奏，响应更稳定。  
**缺点**：具有侵入性，需要轮椅控制器支持此恢复路径。

---

## 5. 启动时模式选择

上电后，持续按住 PS4 按键 **10 秒**：

| 按键 | 选择模式 |
|------|---------|
| □（Square）| FollowJSM（默认） |
| ×（Cross）| JSMerror |

超时未按则自动使用 FollowJSM。

---

## 6. Slot ID 自动探测

启动时在 CAN 总线上监听最多 2 秒，寻找匹配 `ID & 0x0F000000 == 0x02000000` 的帧。  
找到则使用其实际 ID；超时则回退到默认 `0x02000100`（Slot 1）。

---

## 7. 安全看门狗

- 若 PS4 蓝牙连接断开超过 **3000 ms**（`WATCHDOG_TIMEOUT_MS`），立即改为发送停止帧（X=0, Y=0）
- 每 10 ms 发送一次停止帧直至重新连接
- WebSocket 遥测推送频率：**100 ms**（`WS_BROADCAST_INTERVAL_MS`）
- 电池历史采样：**1 秒** 一次（保留最近 60 条）

---

## 8. cansend 格式速查

代码内部使用文本格式 `"XXXXXXXX#YYYYYY"` 简化帧发送：

```
格式：  <8位16进制ID>#<数据字节对>
例：    "0A040100#4B"   → CAN ID 0x0A040100, 数据 [0x4B]（速度 75%）
       "0C040100#"     → CAN ID 0x0C040100, 无数据（喇叭 ON）
       "0C000000#"     → CAN ID 0x0C000000, 无数据（JSM 错误触发帧）
```
